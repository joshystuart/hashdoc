# The secure-link split button

Encryption is initiated from a **split button** rather than an inline checkbox. Copying a Link is a single control with a primary segment and a caret menu: with no password the primary copies the plain Link and the menu offers "Copy secure link"; once a password is set the primary becomes "Copy secure link" (with a closed-lock icon) and the menu offers the plain copy plus "Change password…" and "Remove password". Choosing "Copy secure link" with no password yet opens a `PasswordDialog` (a native `<dialog>`) whose submit saves the password, copies the secure Link, and closes. The same control ships in both the Editor and the Viewer. In the same change the user-facing and internal vocabulary unifies on **"secure"** (the format-v2, password-encrypted scheme previously called "protected").

## Why

The previous Editor surfaced encryption as a "Protect with password" checkbox on its own row, with password + confirm fields that appeared below it and a copy action that silently switched to the encrypted Link once the fields matched. That separates the _decision to protect_ from the _act of copying_, hides the security state from the one button people actually click, and only exists in the Editor — the Viewer had a plain "Copy Link" with no way to express intent. Folding protection into the copy control itself makes the secure variant a first-class, discoverable sibling of the plain copy, keeps the active scheme visible on the button that performs the action (lock = secure), and gives the Viewer the same affordance. Consolidating on one word ("secure") removes the "protected" vs "secure" split that the glossary exists to prevent.

## Considered options

- **Control shape.** Chose a true split button (primary segment + caret menu) over two always-visible buttons or a single relabelling button. The split button keeps exactly one primary action, reflects the active scheme on the segment that copies, and leaves room for the secondary actions (plain copy, change, remove) in the menu without crowding the header.
- **What the primary copies once a password is set.** The primary _swaps_ to "Copy secure link" and the plain copy moves into the menu, rather than the primary staying plain with a lock shown only as an indicator. Rationale: after deliberately setting a password, the safe default is the secure Link; demoting the plain copy to the menu makes accidentally sharing the unencrypted Link a conscious choice, not the default.
- **Subsequent secure copies.** Clicking the now-primary "Copy secure link" copies immediately using the session-remembered password — no modal — matching how plain "Copy Link" already behaves. The modal only appears for the first-time and "Change password…" flows. Re-prompting on every copy was rejected as needless friction.
- **Modal input.** A single password field with a show/hide reveal toggle, not a password+confirm pair. The reveal lets an author verify there is no typo (the failure mode that matters, since a typo yields an unrecoverable Document) while the explicit "Copy secure link" submit button is the deliberate action. The pre-existing confirm field is dropped.
- **Modal submit semantics.** In the first-time flow the submit button reads "Copy secure link" and does all three things — save the password, write the secure Link to the clipboard, close — fulfilling the copy intent in one step. The "Change password…" flow reuses the same dialog but only saves (no copy), since changing a password is not a copy request.
- **Password management.** Once set, the caret menu carries "Copy Link" (plain), "Change password…", and "Remove password". "Remove" is split out from "Change" so the destructive revert (back to no protection) is distinct from editing the secret. Removal is non-destructive of data — it only clears the session password and reverts the button — so it needs no extra confirmation.
- **Viewer seeding.** The Viewer's split button is seeded from how the Document was opened. A plain Link starts with the plain primary and can encrypt the in-memory plaintext on demand exactly like the Editor. A secure Link starts locked: the primary "Copy secure link" re-emits the _original_ Payload byte-for-byte (no re-encryption, no password needed), and the menu still offers the plain copy — no new leak, since the reader already holds the plaintext and "Copy source".
- **Module seams.** A shared `SplitButton` is introduced because two callers (Editor, Viewer) genuinely vary across it — a real seam. A `PasswordDialog` is built directly on the native `<dialog>` element (top-layer, built-in focus trap, ESC-to-close, `::backdrop`) rather than a generic `Modal` shell, because there is only one dialog use today; a generic modal seam would be speculative.
- **Terminology.** Chose to rename "protected" → "secure" everywhere (UI, code identifiers, glossary, `FORMAT.md`, ADR 0003) over keeping "protected" internal or relabelling the button to "protected". A single term across surfaces is the point of the glossary, and "secure" is the word the control already shows users. This is a **language-only** rename: the wire format, the binary frame, and the `2` version tag are unchanged, so every existing secure Link keeps decrypting.

## Threat model

- The split button does not change the cryptography of ADR 0003 — same PBKDF2-HMAC-SHA-256 (600k iterations) + AES-256-GCM, same offline-brute-force bound on a Link-holder, same "lost password is unrecoverable". The safety copy that previously sat under the checkbox now lives inside the `PasswordDialog`, shown at the moment the author chooses the password.
- **Demotion is a deliberate act.** Because the plain "Copy Link" moves into the menu once a password is set, downgrading a secure Document to an unencrypted Link requires opening the menu and choosing it — it cannot happen by clicking the primary button out of habit.
- **The session password lives only in memory** for the life of the Editor/Viewer instance, exactly as today's `password` state did; it is never written to the Link, storage, or anywhere persistent.
- **Unprotected copies make no security claims.** Only the secure segment and the dialog use security language; the plain "Copy Link" stays honestly unprotected, preserving the ADR 0003 guardrail.

## Consequences

- **New shared UI primitives.** `SplitButton` and `PasswordDialog` join `HeaderButton`/`AppHeader` in the web package's chrome. `PasswordDialog` establishes the project's first dialog pattern; future modals should follow its native-`<dialog>` approach.
- **The Editor's inline protect section is removed** — the checkbox, the password/confirm inputs, and the inline note all go; their state collapses into the split button's session password and the dialog. The character-count/size indicator continues to reflect whichever Link the primary action would copy (the secure, larger Link once a password is set).
- **Clipboard writes stay within the user gesture.** Encryption is async (Web Crypto), but the clipboard write must ride the click gesture. With a session password set the secure Link is precomputed (as the Editor already does today) so the primary copy is instant; in the dialog's first-time flow the encrypt-then-write happens inside the submit handler's gesture.
- **The "secure" rename touches several files** (`encodeProtected`/`decodeProtected`/`isProtected`, the `password-required` reasoning, the `protectedPayload` prop, plus prose in `FORMAT.md`, ADR 0003, and `docs/specs/encryption/`). The change is mechanical and the `2` tag is untouched, so it carries no format-compatibility risk; existing tests and golden/freeze fixtures keep passing.
- **The Viewer gains parity with the Editor** for copy intent while still re-emitting the exact original Payload for secure Documents, so a reader re-sharing a secure Link never accidentally produces a downgraded plaintext Link.
